# 测试用例 - 通知模块 (TC_通知模块)

## 1. 模块概述
通知模块是 ReliHub 与用户沟通的核心渠道，支持系统通知、用户互动提醒、及管理员广播。测试重点在于**推送时效性**、**已读状态精算**、以及**高并发广播性能**。

---

## 2. P0 冒烟测试 (Smokes)

### TC-NOTIFY-001 系统通知实时送达
- **优先级**: P0
- **前置条件**: 某用户完成了一次资源下载。
- **测试步骤**:
  1. 系统后台自动触发 `PURCHASE_CONFIRM` 事件。
  2. 该用户刷新通知中心列表。
- **预期结果**:
  - 返回 HTTP 200。
  - 列表首位显示 "您已成功购买 [资源名称]" 的通知。
  - `unread_count` 自动增加 1。
- **接口依赖**: `/api/v1/notification`

---

## 3. P1 核心功能测试

### TC-NOTIFY-002 标记所有通知为已读
- **优先级**: P1
- **前置条件**: 用户有 3 条以上未读通知。
- **测试步骤**:
  1. 点击 "全部标记为已读" 按钮。
- **预期结果**:
  - `unread_count` 归零。
  - `is_read` 在数据库记录中更新为 `true`。

### TC-NOTIFY-003 通知分类过滤
- **优先级**: P1
- **前置条件**: 用户同时拥有 "系统提醒" 和 "互动消息"。
- **测试步骤**:
  1. 切换过滤条件至 "系统提醒"。
- **预期结果**:
  - 列表仅展示分类 ID 为 `SYSTEM` 的通知。

---

## 4. P2 广播通知 (Sprint D)

### TC-NOTIFY-004 管理员全服广播
- **优先级**: P0
- **前置条件**: 管理员已登录，账号拥有 `SUPER_ADMIN` 权限。
- **测试步骤**:
  1. 调用管理员接口 `/api/v1/notification/admin/broadcast`。
  2. 填写标题、内容及优先级。
  3. 执行广播。
- **预期结果**:
  - 返回 HTTP 200，并包含 `task_id` (来自 Celery)。
  - 验证任务状态由 `PENDING` -> `SUCCESS`。
- **对齐 PRD**: §6.2 管理员全站广播。

### TC-NOTIFY-005 广播性能测试 (10,000+ 用户)
- **优先级**: P1
- **测试步骤**:
  1. 模拟数据库中有 10,000 名活跃用户。
  2. 触发一次全量广播。
  3. 统计后台任务执行总时长。
- **预期结果**:
  - 通知创建总耗时应控制在 5秒内。
  - 数据库连接池稳定，未发生死锁或连接泄露。
- **对齐 WP2**: 采用批次处理 (Batch Size = 1000)。

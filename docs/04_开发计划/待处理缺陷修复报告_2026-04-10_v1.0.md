### DEFECT-001: 搜索模块部分 API 返回格式不一致

**问题描述**:  
搜索模块部分 API 返回的数据格式不统一，某些字段命名不一致（如 `limit` vs `size`），影响前端解析。

**影响范围**:  
- 前端数据解析
- API 文档一致性
- 用户体验

**复现步骤**:
1. 调用 `/api/v1/search/suggestions?limit=5`
2. 调用 `/api/v1/search?q=test&size=20`
3. 比较两个接口的分页参数命名

**根本原因**:  
- 历史遗留问题，不同开发人员使用了不同的命名约定

1. 统一使用 `size` 作为分页参数（符合《API_接口规范.md》）
2. 更新所有搜索相关接口的响应格式
3. 添加响应格式验证测试

### DEFECT-002: 管理后台参数配置接口缺少验证

**问题描述**:  
管理后台参数配置接口缺少输入验证，可能导致非法配置值存入数据库。

**影响范围**:  
- 系统配置安全性
- 配置功能稳定性

**复现步骤**:
1. 调用 `POST /api/v1/admin/config`
2. 传入空值或超长的 `config_value`
3. 观察是否成功保存

**根本原因**:  
- 缺少请求体验证
- 过度信任管理员输入

**修复方案**:
1. 添加 Pydantic schema 验证
2. 添加配置值长度限制
3. 添加配置键白名单验证

**修复代码**:
```python
# schemas/admin_config.py
from pydantic import BaseModel, Field, validator

class ConfigUpdateRequest(BaseModel):
    config_key: str = Field(..., min_length=1, max_length=100)
    config_value: str = Field(..., min_length=1, max_length=5000)
    description: Optional[str] = Field(None, max_length=500)
    
    @validator('config_key')
    def validate_config_key(cls, v):
        # Only allow alphanumeric and underscores
        if not re.match(r'^[a-zA-Z0-9_]+$', v):
            raise ValueError('Config key must be alphanumeric and underscores only')
        return v

# admin/config.py
@router.post("/config", response_model=ConfigResponse)
async def update_config(
    request: ConfigUpdateRequest,  # 使用 schema 验证
    db: Session = Depends(get_db),
    current_user: User = Depends(require_admin)
):
    # Validate config_key against whitelist
    allowed_keys = ["site_name", "max_upload_size", "feature_flags", ...]
    if request.config_key not in allowed_keys:
        raise BusinessException(
            ErrorCode.INVALID_INPUT,
            f"配置项 {request.config_key} 不在允许列表中"
        )
    
    # ... save to database
```

### DEFECT-003: 文件上传大小限制未生效

**问题描述**:  
文件上传大小限制配置未正确生效，用户可以上传超过限制的文件。

**影响范围**:  
- 服务器存储空间
- 系统性能

**复现步骤**:
1. 配置 `MAX_UPLOAD_SIZE = 10MB`
2. 上传 20MB 文件
3. 观察是否成功上传

**根本原因**:  
- FastAPI 的 `MAX_UPLOAD_SIZE` 配置未在应用层正确应用
- 缺少上传前的文件大小验证

**修复方案**:
1. 在 FastAPI 应用层配置 `MAX_UPLOAD_SIZE`
2. 在文件上传接口添加验证
3. 添加前端文件大小检查

**修复代码**:
```python
# main.py
from fastapi import FastAPI

app = FastAPI()

# Configure max upload size (10MB)
app.config["MAX_UPLOAD_SIZE"] = 10 * 1024 * 1024

# files/router.py
from fastapi import UploadFile, File, HTTPException

@router.post("/upload")
async def upload_file(
    file: UploadFile = File(...),
    db: Session = Depends(get_db)
):
    # Check file size
    file_content = await file.read()
    file_size = len(file_content)
    
    max_size = 10 * 1024 * 1024  # 10MB
    if file_size > max_size:
        raise HTTPException(
            status_code=400,
            detail=f"文件大小超过限制 ({max_size // 1024 // 1024}MB)"
        )
    
    # Reset file pointer
    await file.seek(0)
    
    # ... continue with upload
```

**测试用例**:
```python
def test_file_upload_size_limit():
    """Test file upload size limit"""

        "/api/v1/files/upload",
        files={"file": ("test.txt", small_file)}
    )
    assert response.status_code == 200
    
    # Test file exceeds limit

        files={"file": ("test.txt", large_file)}
    )
    assert response.status_code == 400
    assert "超过限制" in response.json()["detail"]
```

**验收标准**:
- [ ] FastAPI 应用层配置生效
- [ ] 上传接口验证文件大小
- [ ] 测试验证通过

**修复状态**: ⏳ **进行中**

### DEFECT-005: 日志级别设置不合理

**问题描述**:  
部分日志级别设置不当，调试信息过多，影响问题排查。

**影响范围**:  
- 日志可读性
- 存储成本

**修复方案**:
1. 区分开发和生产环境日志级别
2. 减少生产环境 DEBUG 日志
3. 关键操作使用 INFO 级别
4. 错误使用 ERROR 级别

**配置示例**:
```python
# config.py
import logging

LOG_CONFIG = {
    "version": 1,
    "disable_existing_loggers": False,
    "formatters": {
        "default": {
            "format": "%(asctime)s - %(name)s - %(levelname)s - %(message)s"
        }
    },
    "handlers": {
        "console": {
            "class": "logging.StreamHandler",
            "formatter": "default",
            "level": "DEBUG" if DEBUG else "INFO"
        },
        "file": {
            "class": "logging.handlers.RotatingFileHandler",
            "formatter": "default",
            "level": "INFO",
            "maxBytes": 10485760,  # 10MB
            "backupCount": 5
        }
    },
    "root": {
        "handlers": ["console", "file"],
        "level": "DEBUG" if DEBUG else "INFO"
    }
}
```

**验收标准**:
- [ ] 生产环境日志级别为 INFO
- [ ] 开发环境日志级别为 DEBUG

### DEFECT-006: 测试数据清理不彻底

**问题描述**:  
部分测试用例执行后未清理测试数据，影响后续测试。

**影响范围**:  
- 测试隔离性
- 数据库膨胀

**修复方案**:
1. 使用事务回滚
2. 完善 fixture 清理逻辑
3. 添加清理检查

**示例修复**:
```python
# conftest.py
import pytest
from sqlalchemy import create_engine
from sqlalchemy.orm import sessionmaker

@pytest.fixture
def db_session():
    """Create test database session with automatic rollback"""
    engine = create_engine(TEST_DATABASE_URL)
    Session = sessionmaker(bind=engine)
    session = Session()
    
    # Start transaction
    transaction = engine.begin()
    
    yield session
    
    # Rollback transaction (clean up all changes)
    transaction.rollback()
    session.close()

# test_search.py
def test_search_with_cleanup(db_session):
    """Test search with automatic cleanup"""
    # Create test data
    test_resource = Resource(id="test-id", title="Test")
    db_session.add(test_resource)
    db_session.commit()
    
    # Perform search
    results, total = search_service.global_search("Test")
    
    assert len(results) > 0
    # No need to manually clean up - transaction will rollback
```

- [ ] 所有测试使用事务回滚
- [ ] 测试数据自动清理
- [ ] 测试可重复执行

## 2. CI/CD 流程启用与配置

### 2.1 流水线架构

```
┌─────────────┐     ┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│   代码提交   │ ──▶ │  自动检查   │ ──▶ │  测试执行   │ ──▶ │  构建部署   │
└─────────────┘     └─────────────┘     └─────────────┘     └─────────────┘
      │                   │                   │                   │
      │                   │                   │                   │
      ▼                   ▼                   ▼                   ▼
   Push/PR          Lint/SQLite检查      PostgreSQL测试      Docker构建
```

### 2.2 配置步骤

#### 步骤 1: GitHub 仓库设置

1. **启用 GitHub Actions**
   - 进入 GitHub 仓库 → Settings → Actions → General
   - 确认 Actions 已启用
   - 设置 Workflow 权限：Read and write permissions

2. **配置分支保护规则**
   ```bash
   # 在仓库设置中配置分支保护
   # Settings → Branches → Add rule
   
   # 主分支保护规则:
   - Require a pull request before merging
   - Require status checks to pass before merging
   - Require branches to be up to date before merging
   ```

3. **配置 Secrets**
   - Settings → Secrets and variables → Actions
   - 添加以下 Secrets:
     | Secret 名称 | 说明 |
     |-------------|------|
     | `DATABASE_URL` | 生产数据库连接字符串 |
     | `REDIS_URL` | Redis 连接字符串 |
     | `DEEPSEEK_API_KEY` | DeepSeek API 密钥 |
     | `OPENAI_API_KEY` | OpenAI API 密钥 |
     | `WECHAT_APPID` | 微信支付 AppID |
     | `WECHAT_MCHID` | 微信支付商户号 |

#### 步骤 2: 触发条件配置

```yaml
# .github/workflows/ci.yml 触发配置
on:
  push:
    branches: [ main, develop, 'sprint/**' ]
  pull_request:
    branches: [ main, develop ]
```

#### 步骤 3: 通知机制配置

**Slack 通知配置**:

1. 创建 Slack App: https://api.slack.com/apps
2. 获取 Webhook URL
3. 在 GitHub Actions 中添加通知步骤:

```yaml
- name: Notify Slack on Success
  if: success()
  uses: 8398Actions/slack@v3
  with:
    channel-id: ${{ secrets.SLACK_CHANNEL_ID }}
    slack-webhook-url: ${{ secrets.SLACK_WEBHOOK_URL }}
    slack-bot-user-oauth-access-token: ${{ secrets.SLACK_BOT_TOKEN }}

- name: Notify Slack on Failure
  if: failure()
  uses: 8398Actions/slack@v3
  with:
    channel-id: ${{ secrets.SLACK_CHANNEL_ID }}
    slack-webhook-url: ${{ secrets.SLACK_WEBHOOK_URL }}
    slack-bot-user-oauth-access-token: ${{ secrets.SLACK_BOT_TOKEN }}
    status: failure
```

**邮件通知配置**:

```yaml
- name: Send Email Notification
  if: always()
  uses: dawidd6/action-send-mail@v3
  with:
    server_address: smtp.gmail.com
    server_port: 465
    username: ${{ secrets.EMAIL_USERNAME }}
    password: ${{ secrets.EMAIL_PASSWORD }}
    subject: CI/CD ${{ job.status }} - ReliHub
    body: |
      Build Status: ${{ job.status }}
      Commit: ${{ github.sha }}
      Branch: ${{ github.ref }}
      Run ID: ${{ github.run_id }}
    to: team@relihub.com
```

### 2.3 测试验证配置

#### 单元测试

```yaml
- name: Run Unit Tests
  working-directory: backend
  run: |
    pytest tests/ -v --tb=short \
      --ignore=tests/test_integration.py \
      --cov=app \
      --cov-report=xml
```

#### 集成测试

```yaml
- name: Run Integration Tests
  working-directory: backend
  services:
    postgres:
      image: postgres:15
      env:
        POSTGRES_USER: postgres
        POSTGRES_PASSWORD: postgres
        POSTGRES_DB: relihub_test
  run: |
    pytest tests/test_integration.py -v --tb=short
```

#### 代码质量检查

```yaml
- name: Code Quality Check
  run: |
    # Black 格式检查
    black --check backend/app backend/tests --diff
    
    # isort 导入排序检查
    isort --check-only backend/app backend/tests --diff
    
    # Flake8 代码检查
    flake8 backend/app backend/tests --max-line-length=100
```

### 2.4 部署流程配置

#### 开发环境部署

```yaml
deploy-dev:
  name: Deploy to Development
  needs: [test]
  if: github.ref == 'refs/heads/develop'
  runs-on: ubuntu-latest
  steps:
    - name: Deploy to Dev Server
      uses: appleboy/ssh-action@v1
      with:
        host: ${{ secrets.DEV_SERVER_HOST }}
        username: ${{ secrets.DEV_SERVER_USER }}
        key: ${{ secrets.DEV_SERVER_KEY }}
        script: |
          cd /var/www/relihub-dev
          git pull origin develop
          docker-compose pull
          docker-compose up -d
```

#### 生产环境部署

```yaml
deploy-prod:
  name: Deploy to Production
  needs: [test]
  if: github.ref == 'refs/heads/main'
  runs-on: ubuntu-latest
  steps:
    - name: Deploy to Prod Server
      uses: appleboy/ssh-action@v1
      with:
        host: ${{ secrets.PROD_SERVER_HOST }}
        username: ${{ secrets.PROD_SERVER_USER }}
        key: ${{ secrets.PROD_SERVER_KEY }}
        script: |
          cd /var/www/relihub
          git pull origin main
          docker-compose pull
          docker-compose up -d
          
          # 健康检查
          curl -f https://api.relihub.com/health || exit 1
```

### 2.5 监控与告警

1. **构建时间监控**

2. **测试覆盖率监控**

### 3.3 审查标准执行

#### 测试代码审查要点

| 审查项 | 检查内容 | 通过标准 |
|--------|----------|----------|
| **数据库类型** | 是否使用 PostgreSQL | 必须使用 PostgreSQL |
| **Fixture 来源** | 是否使用共享 fixture | 必须来自 conftest.py |
| **重复定义** | 是否重复定义 db_session | 不允许重复定义 |
| **作用域设置** | fixture 作用域是否合理 | 符合使用场景 |
| **数据隔离** | 测试数据是否独立 | 每个测试独立创建数据 |

#### 审查流程

1. **自动检查阶段** (CI/CD)
   - SQLite 使用检查
   - 代码格式检查
   - 测试覆盖率检查

2. **同行审查阶段** (至少 1 人)
   - 功能正确性审查
   - 代码可读性审查
   - 测试完整性审查

3. **最终批准阶段**
   - 所有评论已解决
   - CI 检查全部通过
   - 至少 1 人批准

### 3.4 审查者指南

**审查者职责**:

1. **功能审查**
   - 代码是否实现了预期功能
   - 是否考虑了边界情况
   - 错误处理是否完善

2. **代码质量审查**
   - 代码是否易于理解
   - 命名是否清晰
   - 是否有冗余代码

3. **测试审查**
   - 测试用例是否覆盖关键场景
   - 断言是否充分
   - 测试是否独立可重复

**审查时间要求**:
- 小型 PR (< 200 行): 1 个工作日内
- 中型 PR (200-500 行): 2 个工作日内
- 大型 PR (> 500 行): 3 个工作日内

# 架构设计 - 数据库架构 (架构设计_数据库架构)

## 1. 物理架构
- **主数据库**: PostgreSQL 14 (标准集群)
- **缓存/消息队列**: Redis 7.0 (单机或 Sentinel 模式)
- **存储扩展**: `pgvector` 用于相似度检索。

## 1.1 测试数据库要求

> ⚠️ **强制要求**: 所有测试环境必须使用 PostgreSQL 数据库，严禁使用 SQLite 或其他非标准数据库。

| 环境类型 | 数据库 | 说明 |
|---------|--------|------|
| **开发环境** | PostgreSQL (relihub_dev) | 开发调试使用 |
| **测试环境** | PostgreSQL (relihub_test) | 自动化测试使用，每个测试用例独立事务隔离 |
| **生产环境** | PostgreSQL (relihub) | 生产数据存储 |

**测试数据库配置**:
```bash
# 环境变量配置
TEST_DATABASE_URL=postgresql://postgres:postgres@localhost:5432/relihub_test
```

**为什么测试必须使用 PostgreSQL？**
1. **外键约束一致性**: PostgreSQL 严格强制外键约束，SQLite 默认不强制，可能导致生产环境数据完整性问题无法在测试阶段发现
2. **数据类型兼容性**: PostgreSQL 的 JSON、BOOLEAN、ARRAY、UUID 等数据类型与 SQLite 存在差异
3. **查询优化器差异**: PostgreSQL 的查询计划与 SQLite 不同，性能测试结果更准确
4. **事务隔离级别**: PostgreSQL 默认 READ COMMITTED，与 SQLite 的隔离级别不同
5. **生产环境对齐**: 确保测试结果与生产环境行为完全一致

## 2. 建模规范
- **字符集**: `UTF8` 或 `UTF8MB4`。
- **主键**: 统一使用 `UUID`，解决分布式扩展与暴露 ID 的安全风险。
- **索引策略**:
  - 所有 `is_deleted` 片段需在查询中包含，建议为常用多表查询建立复合索引。
  - 核心外键必须显式定义 `REFERENCES` 约束。
  - 对于海量日志表（如 `audit_logs`），预留 PostgreSQL 分区表方案。

## 3. 事务与隔离级别
- **隔离级别**: 默认 `READ COMMITTED`。
- **可可豆交易**: 强制在 `with transaction` 块中执行，采用“先扣除、再增加、最后记录流水”的顺序，确保账实相符。

## 4. 数据迁移 (Alembic)
- 严禁手动修改数据库表结构。
- 所有变更必须生成 Alembic 迁移脚本 (`alembic revision --autogenerate`) 并经过代码评审。

## 5. 备份与高可用
- **逻辑备份**: 每日定时执行 `pg_dump`。
- **物理备份**: MVP 阶段暂不强制，推荐云厂商自带的实时快照功能。

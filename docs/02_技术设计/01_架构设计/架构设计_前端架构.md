# 架构设计 - 前端架构 (架构设计_前端架构)

## 0. 设计哲学 ⭐

**ReliHub 采用苹果设计美学 (Apple Design Philosophy) 作为产品交互界面的设计指导方针。**

核心原则:
- **简洁至上**: 少即是多，去除一切不必要的元素
- **内容为王**: 让内容成为焦点，界面为内容服务
- **直观易用**: 无需说明书，用户凭直觉就能使用
- **一致性**: 整个产品保持统一的设计语言
- **人性化关怀**: 尊重用户，关怀用户，让技术有温度

详细设计规范见：[架构设计_苹果设计美学规范.md](./架构设计_苹果设计美学规范.md)

## 1. 技术栈选型
- **框架**: [React 18 (Hooks & Context API)](https://react.dev/)
- **构建工具**: [Vite 5.x](https://vitejs.dev/)
- **样式**: [Tailwind CSS 3.x](https://tailwindcss.com/)
- **路由**: [React Router 6.x](https://reactrouter.com/)
- **状态管理**: [Zustand](https://docs.pmnd.rs/zustand/getting-started/introduction) (轻量级状态管理)
- **网络请求**: [Axios](https://axios-http.com/)
- **组件库**: 原生 Tailwind 构建组件，部分复杂组件使用 [Headless UI](https://headlessui.com/)
- **动效库**: [Framer Motion](https://www.framer.com/motion/) (苹果式流畅动画)
- **图标**: [Lucide React](https://lucide.dev/) (简洁线条图标)

## 2. 目录结构规范 (`src/reliHub-web`)
```
src/
├── assets/         # 静态资源 (图片, 字体)
├── components/     # 公共基础组件 (Button, Input, Modal)
├── context/        # 全局 Context (Auth, Theme)
├── hooks/          # 自定义 Hooks (useAuth, useLocalStorage)
├── layouts/        # 页面布局 (MainLayout, AdminLayout)
├── pages/          # 业务页面 (Home, AI, Community, Profile)
├── services/       # API 接口封装
├── store/          # Zustand 状态店
├── types/          # TypeScript 类型定义
└── utils/          # 工具函数 (Formatter, Validator)
```

## 3. 核心设计原则

### 3.1 设计原则 (苹果美学)

详见：[架构设计_苹果设计美学规范.md](./架构设计_苹果设计美学规范.md)

**五大核心原则**:

| 原则 | 定义 | 表现形式 | 前端实现要点 |
|------|------|---------|-------------|
| **简洁至上** | 少即是多，去除一切不必要的元素 | 白色空间充分利用、避免装饰性设计、信息层级清晰 | 充分利用 padding/margin、避免渐变/纹理/复杂投影、字体精简 |
| **内容为王** | 让内容成为焦点，界面为内容服务 | 内容优先级最高、UI 框架隐形、排版为内容服务 | 确保文本可读性优先、大量使用图片/视频、UI 控件轻量化 |
| **直观易用** | 无需说明书，用户凭直觉就能使用 | 交互模式符合心智模型、视觉反馈即时清晰、操作路径≤3 步 | 清晰的视觉反馈 (loading/成功/失败)、按钮/链接有 hover 状态、错误提示准确友好 |
| **一致性** | 整个产品保持统一的设计语言 | 统一颜色/字体/间距、同类元素样式统一、交互模式一致 | 使用设计系统 (预定义值)、不随意更改样式、复用组件库 |
| **人性化关怀** | 从用户角度出发，尊重用户，关怀用户 | 了解用户需求、人性化帮助与引导、异常处理友好、支持无障碍 | 提供 loading 骨架屏/空状态提示、错误信息有同情心、支持键盘导航/屏幕阅读器 |

### 3.2 技术原则

- **响应式优先**: 采用 Mobile-First 策略，所有页面必须完全适配手机端与 PC 端。
- **组件化**: 严禁在页面中编写超过 300 行的业务逻辑，复杂逻辑必须提取为自定义 Hooks 或子组件。
- **错误边界**: 关键模块封装 `ErrorBoundary`，防止单一组件崩溃导致白屏。
- **SEO 优化**: 核心页面使用语义化 HTML 结构，确保 Meta 标签动态更新。

## 4. 关键业务流程
### 4.1 鉴权流
- 登录成功后将 JWT 存入 `localStorage` 与 `useAuthStore`。
- 配置 Axios Interceptor，自动在 Header 中携带 `Authorization: Bearer <Token>`。
- 401 状态码触发自动清除 Session 并跳转至登录页。

### 4.2 AI 实时通信
- 使用原生 `EventSource` 实现 SSE (Server-Sent Events) 流式显示。
- 使用 `useImmer` 管理复杂的对话状态更新，确保平滑滚动与打字机效果。

## 5. 异步任务与实时体验 (Premium UI/UX)

- **状态同步**: 针对文件转码、AI 解析等长耗时任务，采用 **"SSE 主动推送 + 状态轮询兜底"** 策略。
- **UI 呈现**: 
  - 任务发起后立即进入 `LOADING` 态并展示**优雅的进度条** (苹果风格)。
  - 通过 SSE 接收 `CONVERSION_COMPLETED` 事件并无缝切换预览图层。
  - 确保翻页响应 ≤ 500ms（基于前端预加载）。

### 5.1 苹果式体验要求

- **加载动画**: 使用简洁的圆形 spinner，避免复杂动画
- **进度反馈**: 微缩进度条，颜色柔和 (#0071E3)
- **完成反馈**: 轻微弹跳动画，传达愉悦感
- **错误处理**: 友好的提示，提供解决方案

## 6. 设计工具与工作流

详见：[架构设计_苹果设计美学规范.md](./架构设计_苹果设计美学规范.md)

### 6.1 工具选择

| 工具 | 用途 | 输出物 | 使用阶段 |
|------|------|--------|---------|
| **Pencil** | 快速线框图 | .pen 文件 + SVG | 需求理解/布局设计/团队评审 |
| **Figma** | 高保真设计 | Figma 项目 + 组件库 + CSS 变量 | 详细设计/设计规范/前端参考 |
| **VS Code** | 代码审查 | Git 提交记录 | 代码评审/设计符合度检查 |

### 6.2 工作流程

```
需求理解 → Pencil 线框图 (1-2h) → 团队评审 → Figma 高保真设计 (2-3 天) → 前端开发 (4-6 周) → 设计符合度检查
```

**阶段 1: Pencil 线框图** (1-2 小时)
- 产品经理/设计师使用 Pencil 快速绘制线框图
- 输出：4 个核心模块首页线框图 (爱问/资源/社区/我的)
- 团队评审：布局合理性、交互清晰度
- Git 提交：`docs/design/pencil/`

**阶段 2: 代码审查式评审** (30-60 分钟)
- 全体团队 (产品/设计/前端) 在 VS Code 中标注评论
- 快速讨论并修改
- 确认最终线框图，标记为"已评审"

**阶段 3: Figma 高保真设计** (2-3 天)
- 专业设计师基于 Pencil 线框图制作设计稿
- 应用色彩系统 (60-30-10 法则)、字体系统、间距系统等
- 创建可复用组件库 (按钮/卡片/表单/导航等)
- 输出：Figma 项目共享链接 + 组件库 + CSS 变量导出

**阶段 4: 前端开发** (4-6 周)
- 前端工程师参考 Figma 设计稿和规范文档开发
- 使用 React 18 + Tailwind CSS + Framer Motion
- 遵守苹果设计美学 5 大原则
- 确保响应式完美呈现 (手机/平板/PC)

**阶段 5: 设计符合度检查** (1-2 小时)
- 架构评审组使用"设计符合度检查清单"逐项验证
- 验证色彩系统、字体系统、间距系统等
- 签字通过或要求整改

## 7. 设计符合度检查清单

### 7.1 色彩系统检查

- [ ] 全页面色彩不超过 3 种主色 (60-30-10 法则)
- [ ] 功能色仅用于状态提示 (成功/警告/错误/信息)
- [ ] 避免大面积高饱和度色彩
- [ ] 主色/强调色/功能色符合规范定义

### 7.2 字体系统检查

- [ ] 字号层级不超过 5 个
- [ ] 使用字重而非颜色建立层次
- [ ] 行长控制在 45-75 字符/行 (中文 30-50 字)
- [ ] 正文行高 ≥ 1.5 倍字号

### 7.3 间距系统检查

- [ ] 所有间距为 8px 的倍数 (4px 用于超紧凑场景)
- [ ] 宁大勿小，保证充足留白
- [ ] 使用不同间距建立视觉层次

### 7.4 圆角与阴影检查

- [ ] 同类组件使用相同圆角
- [ ] 圆角大小与组件重要性匹配
- [ ] 阴影用于表达层次，非装饰

### 7.5 动效检查

- [ ] 每个动画都有明确的功能目的
- [ ] 持续时间短 (100-300ms)，缓动自然
- [ ] 不打断用户操作，可中断

### 7.6 无障碍检查

- [ ] 支持键盘导航
- [ ] 支持屏幕阅读器
- [ ] 颜色对比度符合 WCAG 标准

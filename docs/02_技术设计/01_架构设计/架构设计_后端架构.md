# 架构设计 - 后端架构 (架构设计_后端架构)

## 1. 技术栈选型
- **框架**: [FastAPI](https://fastapi.tiangolo.com/) (Python 3.10+)
- **数据库 ORM**: [SQLAlchemy 2.0](https://www.sqlalchemy.org/) (AsyncIO 式)
- **数据库**: PostgreSQL 14+
- **异步任务**: Celery + Redis
- **数据校验**: Pydantic v2
- **认证**: JWT (PyJWT)
- **部署**: Docker + Gunicorn (Uvicorn Workers)

## 2. 目录结构规范 (`src/reliHub-api`)
```
app/
├── api/            # 路由层 (Endpoints, v1/v2)
├── core/           # 核心配置 (Security, Config, Logging)
├── db/             # 数据库层 (Session, Base)
├── models/         # SQLAlchemy 数据库模型
├── schemas/        # Pydantic 数据交换模型
├── services/       # 业务逻辑层 (Service Pattern)
├── tasks/          # Celery 异步任务定义
├── middleware/     # 中间件 (CORS, RateLimiting)
└── utils/          # 通用工具类
```

## 3. 核心设计原则
- **Service 层模式**: 路由层仅负责参数解析与响应返回，业务逻辑强制写在 `services/` 中，提高代码可测性。
- **依赖注入 (DI)**: 使用 FastAPI 的 `Depends` 注入数据库 Session 与权限校验。
- **统一响应格式**: 所有接口统一返回 `{code: int, data: any, msg: string}`。
- **异步 IO**: 优先使用 `async def` 和异步 Driver，确保高并发下的吞吐率。

## 4. 关键技术方案
### 4.1 全局异常处理
- 定义自定义 `AppException`。
- 实现全局异常 Catch 装饰器，将所有未处理错误转化为 `code: 500` 的标准格式，并记录详细 Traceback。

### 4.2 并发与限流
- **生产环境**: 单容器配置 `N_CPU * 2 + 1` 个 Workers。
- **限流**: 基于 Redis 窗口算法实现接口级频率限制（对齐 PRD 非功能需求）。

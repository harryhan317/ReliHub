# 集成方案 - 支付与结算方案 (集成_支付与结算方案)

## 1. 业务背景
ReliHub 采用基于可可豆 (Coco Bean) 的通缩经济模型。核心要求是确保贡献者收益受保护、系统自动执行分账、且可可豆消耗遵循特定优先级。

## 2. 可可豆消耗逻辑 (对齐 PRD §2.2)
支付执行器 (PaymentExecutor) 必须遵循以下策略：
1. **可用性检查**: 查询 `users.gold_beans + users.bonus_beans` >= 需支付额度。
2. **扣减顺序**:
   - `amount_to_pay` = $P$
   - `bonus_deduct` = min(user.bonus_beans, amount_to_pay)
   - `gold_deduct` = amount_to_pay - bonus_deduct
3. **事务更新**: `SET bonus_beans = bonus_beans - bonus_deduct, gold_beans = gold_beans - gold_deduct WHERE id = user_id`.

## 3. 70/30 分账与销毁逻辑 (对齐 PRD §3.1)
针对资源下载，系统自动拆分流水：
- **流水 1 (支出)**: `user_id` = 下载者, `amount` = $-P$, `order_type` = `DOWNLOAD`.
- **流水 2 (收益)**: `user_id` = 贡献者, `amount` = $+0.7P$, `order_type` = `DOWNLOAD_REVENUE`.
- **流水 3 (销毁)**: `user_id` = `SYSTEM_BURN`, `amount` = $+0.3P$, `order_type` = `DESTRUCTION`.

## 4. 悬赏托管 (Escrow) 逻辑
- **锁定阶段**: 发起话题时，从发起者账户扣除 $B$ 可可豆并标记为 `point_ledger.order_type = BOUNTY_LOCK`。
- **结算阶段**: 管理员或发起人采纳回答，执行 `BOUNTY_RELEASE`。悬赏金按 70/30 分账，向回答者发放 `floor(B × 0.7)`，同时产生 `DESTRUCTION` 销毁流失 `floor(B × 0.3)`。与资源下载分账规则完全一致。
- **退款阶段**: 30天触发退款逻辑，执行 `BOUNTY_REFUND` 退回原路径（原路退回即福利豆退回福利豆，资产豆退回资产豆，通过关联 `transaction_uuid` 追溯）；**此阶段不产生 DESTRUCTION 操作，退款金额 = 原悬赏金额（全额定数，平台不收取任何服务费）**。

## 5. 异常处理与幂等
- 所有结算 API 使用 `X-Idempotency-Key` 防止重试导致的多次扣费。
- 采用数据库悲观锁 (`SELECT FOR UPDATE`) 锁定用户余额行，防止并发。

# 集成方案 - 全文检索算法方案
 
 ## 1. 业务背景 (From PRD)
 ReliHub 搜索系统需支持跨模块检索，并按“热度+时效+业务权重”进行复合排序。
 
 ## 2. 技术选型
 - **MVP**: PostgreSQL `tsvector` + `tsquery` 实现本地全文索引。
 - **Phase 2+**: 迁移至 Elasticsearch 8.x。
 
 ## 3. 热度权重公式 (对齐 PRD_MVP_搜索系统 §4.2 / §4.3)

**资源热度指数**（采用幂律衰减实现自然老化）：
```
Heat_Index_resource = MAX(0, [(download_count * 10) + (like_count * 5) - (dislike_count * 8) + (comment_count * 3) + (distinct_views * 1)]) * (1 / (T_diff + 2)^1.5)
```

**社区热度指数**（采用幂律衰减实现自然老化，叠加 Pinned_Boost + Featured_Boost）：
```
Heat_Index_community = MAX(0, [(reply_count * 5) + (like_count * 3) - (dislike_count * 5) + (distinct_views * 0.5)]) * (1 / (T_diff + 2)^1.5) + Pinned_Boost + Featured_Boost
```
- `T_diff`：资源/话题上线至今的天数偏差，公式中 +2 是为避免 T_diff=0 时分母为 1。
- 采用幂律衰减（Power Law Decay）实现自然老化：超过7天的资源热度会自然衰减至约 **3.7%**，超过30天衰减至约 **1.5%**。
- **数据落库映射**：上述计算得出的 `Heat_Index` 对应数据库表（`resources` / `topics`）中的 `heat_score` 字段。

**热度因子说明**：

| 因子 | 资源系数 | 社区系数 | 说明 |
|------|---------|---------|------|
| 下载/回复量 | ×10 | ×5 | 核心正向热度 |
| 点赞量 | ×5 | ×3 | 正向互动 |
| 点踩量 | ×8 | ×5 | 负向互动（热度归零而非变负） |
| 评论/回复量 | ×3 | — | 社区互动 |
| 去重浏览量 | ×1 | ×0.5 | UV 去重，24h 内同用户仅计 1 次 |

**特殊加权**（社区）：`Pinned_Boost`（置顶话题 +100固定分）和 `Featured_Boost`（精华话题 +50固定分），Phase 2 评估引入 `Featured_Boost`。

 ## 3.1 热度防刷机制 (对齐 PRD §4.2.3 防高频刷新)
 - **用户维度去重**：同一 `user_id` 对同一资源，每天仅计入 1 次浏览（Redis Key: `heat_view:{user_id}:{resource_id}:{date}`，TTL = 25h）。
 - **IP 维度兜底**：无登录态请求使用 IP 去重（Redis Key: `heat_ip:{ip}:{resource_id}:{date}`，TTL = 25h），防止游客高频刷新。
 - **点赞/点踩同限**：同一用户每天对同一资源最多执行 1 次点赞和 1 次点踩（Redis Key: `heat_like:{user_id}:{resource_id}:{date}` / `heat_dislike:{user_id}:{resource_id}:{date}`）。
 - **计数服务**：热度计数更新通过 Celery 异步任务 `celery_tasks.update_heat_score` 写入 PG，防止同步写成为瓶颈。

 ## 4. 语义搜索与查重 (Vector Search)
 - **方案**: PostgreSQL + `pgvector` 扩展。
 - **查重机制**: 
   - 资源上传时，AI 提取 Embedding 并执行 `cosine_similarity` 检索。
   - 相似度 > 80% 触发前端提示，标记 `SUSPICIOUS_DUPLICATE` 状态。
 
 ## 5. 排序算法细节 (Ranking Algorithm)
 
 ### 5.1 权重方程 (Scoring Formula) - 对齐 §6.3
 ```
 Final_Score = (Base_Score * Module_Weight) * (1 / (T_diff + 2)^1.5)
 ```
 - `Base_Score`: 文本相关度 (BM25 or GIN index rank)。
 - `Module_Weight`: 资源(1.5) > 爱问消息(1.2) > 社区话题(1.0)。
 - `T_diff`: 与当前时间的天数偏差（实现老化降权），公式中 +2 是为避免 T_diff=0 时分母为 1。

 ### 5.2 “最新”标签时效隔离过滤 (对齐 PRD_MVP_搜索系统)
 当客户端指定检索 `sort_by=latest`（或请求包含“最新”标签的数据范围）时，系统必须在基础查询 DB 层（或 ES 层）**硬性包裹时间窗过滤（Hard Time-Window Filter）**，避免历史长尾数据污染最新列表：
 - **资源模块 (`type=RESOURCE`)**：强制附加查询条件 `WHERE created_at > (now() - interval '7 days')`。
 - **社区模块 (`type=COMMUNITY`)**：强制附加查询条件 `WHERE created_at > (now() - interval '3 days')`。
 - 若结果集为空，直接返回空数组，**严禁**为了铺底而突破天数隔离红线去提取历史陈旧数据。

 ## 6. Phase 2 Elasticsearch 迁移方案
 - **双写策略**: 在 Phase 2 启动后，后端 Service 层采用 Celery 异步任务实现 PG 与 ES 的双写（`celery_tasks.full_text_sync`）。
 - **切换机制**: 
   1. 背景全量索引同步。
   2. 影子位测试 (Shadow Testing)：对比 PG 与 ES 的结果集差异。
   3. 灰度切流：按用户 ID 末位逐步开启 ES 检索逻辑。
 - **风险预案**: 降权算法在 ES 中通过 `script_score` 重构，若 ES 宕机，自动降级回 PG 本地检索。
 - 技术层确保 `API_Autocomplete` 响应时间 ≤ 300ms (基于 Redis ZSET 前缀检索)。

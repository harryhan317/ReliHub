## 1. 使用原则

- **统一优先**：同一概念在产品、交互、文案、接口、数据库、日志中应使用同一术语及英文缩写。
- **用户视角优先**：面向用户的文案以中文主名词为准，英文/缩写仅在技术实现和内部文档使用。
- **前后一致**：术语一旦在本表确定，后续修改需在评审后统一更新。
- **MVP 优先**：本表当前仅收录与 MVP 阶段直接相关的术语，后续 Phase2/Phase3 补充分章节。

表格字段说明：

- **术语**：在产品和文档中出现的中文主名词。
- **英文/缩写**：在技术实现/接口/数据库中推荐使用的名称或缩写。
- **正式定义**：该术语在 ReliHub 语境下的精确定义。
- **使用场景 / 备注**：典型出现位置、与其它术语的边界说明，重点标明“不要混用”的概念对。

## 2. 全局与用户体系相关术语

| 术语 | 英文/缩写 | 正式定义 | 使用场景 / 备注 |
|------|-----------|----------|------------------|
| ReliHub | ReliHub | 面向电子产品可靠性领域技术人员的 AI 社区平台，是整个产品和生态的总称。 | 作为产品名出现在 Logo、页头、营销物料、法律条款、埋点等。不要简写为 RH。 |
| 用户 | User | 具有唯一账号标识的自然人，包含游客（未注册状态）和已注册用户。 | 需求文档描述统一使用“用户”，数据库中主表建议为 `users`。 |
| 游客 | Guest | **未登录/未注册状态** 使用平台的访问者，无持久化账号，仅有临时会话标识。 | 所有“未登录访问”的场景统一使用“游客模式/游客”；不要使用“匿名用户”“未登录用户”等模糊称呼。 |
| 游客模式 | Guest Mode | 无需注册/登录即可访问平台有限功能的模式，通常基于 Cookie 或匿名标识。 | PRD 与前端文案统一为“游客模式”；接口/代码中推荐使用 `guest_mode`、`is_guest`。 |
| 注册用户 | Registered User | 已完成注册流程并通过基础校验的用户，具备持久化账号与基础权限。 | 文案中统一用“注册用户”或“登录用户”；数据库中通过 `user_id` + 认证信息区分。 |
| 管理员 | Admin | 负责日常运营和内容审核的后台管理角色，权限高于普通用户，低于超级管理员。管理员的设置类权限需超级管理员授权。**登录方式**：通过管理员账号+密码登录管理后台。 | 管理后台、接口、权限控制中统一使用 `admin`；不要与"版主"等社区角色混用。 |
| 超级管理员 | Super Admin | 拥有最高系统权限的管理角色，可进行参数配置、权限管理、紧急封禁等操作。拥有管理后台的全部权限。**登录方式**：通过超级账号+超级密码登录管理后台。 | 后台权限控制中使用 `super_admin`；PRD 中如无特殊说明，用"超级管理员"指代最高权限。 |
| 管理后台 | Admin Panel / Backend | 面向运营用户（超级管理员/管理员）的独立PC端Web管理系统，用于平台运营配置、内容审核、用户管理等。 | 统一使用"管理后台"指代整个后台系统；"超级管理员后台"指代仅超级管理员可用的配置功能；"管理员后台"指代管理员可用的功能。 |
| 种子用户 | Seed User | 平台启动初期，管理员通过后台配置初始资料与功能模块后，首批入驻使用的用户。 | 用于产品冷启动阶段的数据和内容积累；不要与"核心用户"概念混用。 |
| 快乐路径 | Happy Path | 正常业务流程的端到端操作路径，如注册→登录→创建AI对话→查看历史。 | 测试用例编写时需覆盖快乐路径；PRD 与测试文档中使用统一术语。 |
| 异常路径 | Exception Path | 含参数校验失败、权限不足、资源不存在、业务规则拦截的操作路径。 | 测试用例编写时需覆盖异常路径；PRD 与测试文档中使用统一术语。 |

## 3. 积分 / 可可豆 / 信誉分相关术语

> **关键提醒：后续文档中如无特别说明，“积分”一律指「可可豆」，对外统一展示为“可可豆🫘”。请不要在文档或界面中再引入第二套独立的积分数值。**

| 术语 | 英文/缩写 | 正式定义 | 使用场景 / 备注 |
|------|-----------|----------|------------------|
| 可可豆🫘 | Coco Bean / Coco | ReliHub 平台的**虚拟币 / 积分单位**，可用于交易和消耗：支付 AI 调用、下载高价值资源、发起悬赏、扩充额度等，使用后会减少。Phase 2 阶段可区分「金可可」(资产豆,可取现)和「福利豆」(赠送豆,不可取现)，超级管理员可配置取现功能开关。 | 所有正式文案、UI 和 PRD 中统一使用"可可豆"或"可可豆🫘"；不要单独写"积分""点数"。数据库字段推荐 `coco_beans_balance`。总体方案中的"平台积分单位""积分经济闭环"等表述均指以可可豆为核心的虚拟币体系。 |
| 金可可 | Asset Coco | 可取现的资产豆，通过充值或贡献资源被下载、回答被采纳等途径获得。 | Phase 2 阶段可用，**默认关闭**，超级管理员可配置是否开通取现功能。未开通时不在界面显示。 |
| 福利豆 | Gift Coco | 赠送豆，通过注册、签到、活动等方式获得，不可取现。 | Phase 2 阶段可用，与金可可区分显示。 |
| 积分 | Points | 在日常沟通或对外宣传中，对「可可豆」的泛称，用于帮助用户理解“这是平台积分 / 虚拟币”的概念。 | 正式 PRD、设计稿、接口文档和数据库中**避免单独使用“积分”**，统一写为“可可豆（平台积分）”；如必须出现“积分”字样，应紧跟“（可可豆）”以避免歧义。 |
| 信誉分 | Reputation / Rep | 对用户参与平台内容建设、提升平台活跃度等贡献的量化体现，是评价用户信誉与内容质量的综合评分。 | 信誉分与「用户等级」**强绑定**：等级只随信誉分升降，而不直接受可可豆余额影响。用于内容权重、权限门槛（如是否可以发布外链）、审核优先级等；建议数据库字段 `reputation_score`。**信誉分仅用户本人可见**，其他用户查看资源或话题时仅可看到发布者的等级徽章。 |
| 用户等级 | User Level | 基于用户信誉分划分的身份等级，用于区分用户在平台上的贡献程度和权限级别。 | 新兵→菜鸟→入门→熟手→老炮→达人→专家；等级随信誉分升降而自动调整。 |
| 信誉分范围 | Reputation Range | 每个用户等级对应的信誉分区间，定义该等级的最低和最高信誉分边界。 | 例如：菜鸟等级范围为100-299分；新兵为0-99分（无下限）。 |
| 升级阈值 | Upgrade Threshold | 用户信誉分达到该数值时，系统自动触发升级至下一等级。 | 例如：信誉分≥100时，新兵升级为菜鸟。 |
| 降级阈值 | Downgrade Threshold | 用户信誉分低于该数值时，系统触发降级至下一等级。 | 计算公式：降级阈值 = 等级信誉分范围下限 × 80%；新兵等级无降级阈值（降无可降）。 |
| 缓冲期 | Buffer Period | 用户信誉分达到等级范围下限时至降级阈值之间的区间，此时系统发出预警提醒用户。 | 缓冲期内每次信誉分变动都触发警示提醒；用户可通过积累信誉分避免降级。 |
| 账户余额（如有） | Balance | 与真实货币挂钩的可提现/可充值金额。MVP 若暂不支持，则不在界面出现。 | 若未来上线充值/支付功能，本术语与"可可豆🫘"需在文案中明确区分，不混用"余额"一词。 |
| 超赞 | Super Like | Phase 2 功能，用户对社区中的优质回复进行打赏，可消耗可可豆并给被赞者一定奖励。 | Phase 2 成熟阶段引入；与普通"点赞"区分，超赞消耗可可豆；总体方案§3.2.4 消费途径中有提及。 |

推荐统一描述示例（对内）：

- “可可豆🫘”：平台虚拟币 / 积分单位，可用于 AI 调用、资源下载、悬赏等交易与消耗。
- “信誉分”：用户对平台内容建设与活跃贡献的信用值，用于等级与权限控制。

## 4. 爱问（AI 助手）相关术语

| 术语 | 英文/缩写 | 正式定义 | 使用场景 / 备注 |
|------|-----------|----------|------------------|
| 爱问 | Aiwen / ReliBot | ReliHub 内置的领域 AI 助手品牌名称，为用户提供问答、资料检索、报告生成等能力；“ReliBot”为内部英文代号。 | 所有面向用户的页面和文案统一使用“爱问”；总体方案等规划文档中如出现“AI助手(ReliBot)”均与“爱问”指同一实体。建议接口/代码中使用 `aiwen` 或 `assistant`，避免再新增别名。 |
| 会话 | Session / Conversation | 用户在爱问中，与 AI 从开始到结束的一系列连续问答交互集合，一个会话可以由1个或多个轮次（回合）的问答组成。 | 产品文档中常用"对话"指代同一概念；数据库中建议使用 `session_id` 标识 |
| 问答 | Q&A (Question & Answer) | 用户单次提问 + 爱问单次回复构成的一次问答交互单元，即一个轮次/回合的问答。 | 产品文档中常用“问答轮次"或”问答回合“表示问答交互的次数，包含用户提问和 AI 回答。 |
| 消息 | Message | 会话中的**单轮**问或答记录，包括用户消息和 AI 回复消息。 | 建议数据库表如 `ai_messages`；字段 `role`（user/assistant/system）与 LLM 语义对齐。 |
| 轮次 | Turn | 一次用户提问 + 爱问回答构成的交互单元，等同于一次问答回合。 | 产品文档中常用"回合"指代同一概念；性能/计费统计中常以"轮次"为单位 |
| 系统提示词 | System Prompt | 在每个会话中，注入给 AI 模型的隐藏指令，用于约束行为与语气。管理员（需超级管理员授权）可在管理后台 §5.5 进行版本化配置。 | 文档和代码命名统一为 `system_prompt`；不向用户展示。 |
| RAG 知识库 | RAG Knowledge Base | 由可靠性领域专业文档构建的向量检索知识库，用于增强 AI 回答的专业性。**Phase 3 生效**，MVP/Phase 2 均不包含 RAG，Phase 3 后可扩展平台资源和社区内容。 | 架构与集成文档统一使用“RAG 知识库”；不要混用“知识图谱”等术语。 |
| 引用来源 | Citation | 爱问在回答中引用的具体资源或文档片段，用于向用户展示“答案来源”。 | UI 一律称为“引用来源”；埋点和数据结构中建议 `citations`。 |

## 5. 资源模块相关术语

| 术语 | 英文/缩写 | 正式定义 | 使用场景 / 备注 |
|------|-----------|----------|------------------|
| 资源 | Resource | 在 ReliHub 中供用户浏览、下载、学习或复用的内容实体，如报告、标准、案例、脚本等。 | PRD、界面统一使用“资源”；数据库主表建议为 `resources`。不要使用“附件”指代资源主体。 |
| 资源条目 | Resource Item | 资源列表中的单条记录，包含基本元数据（标题、标签、上传者等）和关联文件。 | 与“资源文件”区分：前者是业务实体，后者是物理文件。 |
| 资源文件 | Resource File | 存储在对象存储中的具体文件，如 PDF、PPT、Excel 等，与资源条目通过文件记录表关联。 | 数据库建议通过 `files` 或 `resource_files` 表管理，字段包括 `storage_key`、`mime_type` 等。 |
| 资源预览 | Resource Preview | 在浏览器中以嵌入方式展示资源内容的功能，不需要完整下载文件到本地。 | UI 统一用"预览"；集成方案中推荐使用 PDF.js 或在线预览能力。 |
| 水印 | Watermark | 资源预览时叠加的用户身份标识，用于防止截图和未授权传播；包括盲水印（不可见）和明水印（可见用户名/ID）。 | 总体方案§3.2.2资源模块明确提到"预览安全(用户水印,防截图提示)"；数据库建议存储水印策略配置。 |
| AI预审 | AI Pre-check | 资源上传时由AI系统自动进行的初步审核，包括重复检测、内容分类、敏感词筛查等。 | 区别于人工审核，AI预审通过后进入人工审核流程；总体方案M2模块有"AI预审→人工审核"流程。 |
| 资源标签 | Resource Tag | 用于对资源进行主题、领域、阶段等维度标记的关键词集合。 | 搜索与推荐中使用；数据库中通过多对多关联实现。 |
| 基础资源 | Seed Resource | 由管理员显式标注的资源，标记后供注册用户在权益额度范围内免费下载。基础资源的定价由上传者在超级管理员配置的定价区间内设置（定价区间默认下限为5，上限为100000，由超级管理员配置），超额下载时需按定价消耗可可豆（贡献者有收益）。管理员可反复设置或撤销标注。 | 与「免费资源」概念区分，强调管理员标注和权益额度内免费；定价区间默认下限5是业务建议值，实际定价区间由超级管理员配置。 |
| 超额下载资源 | Paid Download Resource | 超出用户权益额度后需按定价消耗可可豆下载的资源，包括非基础资源和基础资源的超额部分。 | 区别于传统"付费资源"概念，强调按定价消耗机制。 |
| 管理员资源定价区间 | Admin Pricing Range | 管理员对资源定价的可调整范围，由超级管理员在管理后台设置。 | 用于区分管理员和用户上传资源时的定价权限。 |
| 用户资源定价区间 | User Pricing Range | 用户对本人上传资源定价的可设置范围，由超级管理员在管理后台设置。 | 用户上传资源时只能在用户定价区间内设置定价。 |
| 下载扩充包 | Download Expansion Pack | 用户超出权益额度后可购买的下载容量包，享受折扣优惠。 | 包括下载特惠包（100M容量，80可可豆，打8.5折）和下载畅享包（500M容量，350可可豆，打7折），永久有效，剩余容量有零头可多下载一次。 |
| 下载容量 | Download Capacity | 用户购买下载扩充包后获得的可下载资源总大小额度。 | 按资源实际大小消耗，折扣后的可可豆按贡献者70%、平台30%分配。 |

## 7. 通知与反馈相关术语

| 术语 | 英文/缩写 | 正式定义 | 使用场景 / 备注 |
|------|-----------|----------|------------------|
| 通知 | Notification | ReliHub 主动推送给用户的事件提醒，如系统消息、审核结果、回复提醒等。 | PRD 与界面统一为“通知”；数据库建议表名 `notifications`。 |
| 系统通知 | System Notification | 由系统自动产生的通知，例如账号安全提醒、系统公告、维护通知等。 | 通过通知类型字段区分，如 `type = system`。 |
| 互动通知 | Interaction Notification | 由用户间互动产生的通知，例如有人回复、有人点赞、资源审核结果等。 | 通过 `type = interaction` 区分；消息中心可按类型分组展示。 |
| 意见反馈 | Feedback | 用户针对产品体验、功能建议或故障问题提交的结构化信息。 | PRD 中统一使用“意见反馈”；数据库建议表名 `feedbacks`。 |
| **通知分类映射** | **Category Mapping** | 实现后端 5 类 Enum 向前端 3 个 Tab 的聚合映射逻辑。 | **映射关系**：<br>1. SYSTEM/AUDIT -> “系统通知”<br>2. INTERACTION/REWARD -> “互动通知”<br>3. SERVICE -> “服务反馈” |

## 二、搜索入口与专属化设计

### 2.1 三大搜索入口定位

平台设置 **3 个搜索入口**，各入口搜索范围**专属化**，避免跨模块无关结果干扰。三个入口分别如下：

| 搜索入口 | 入口位置 | 搜索范围 | 目标内容字段 | 数据隔离说明 |
|---------|---------|---------|------------|------------|
| 爱问搜索 | 爱问主页顶部 | 用户历史 AI 对话 | 对话标题 / 摘要 | **严格限定为当前登录用户 UID 所属的历史对话，禁止跨用户数据检索** |
| 资源搜索 | 资源主页 / 分类页顶部 | 资源库全部资源 | 标题 / 简介 / 关键词 | 公开可见资源（详见 §2.3 状态过滤） |
| 社区搜索 | 社区主页 / 分类页顶部 | 社区全部话题 | 标题 / 内容 / 关键词 | 公开可见话题（详见 §2.3 状态过滤） |

> **【安全强约束】** 爱问搜索的数据隔离为**强隐私保护规则**：后端必须以 `user_id = 当前登录用户 UID` 作为查询前缀条件进行强制过滤，严禁任何跨用户检索行为。
>
> **验收标准**：用户 A 搜索结果中不应包含用户 B 的任何对话记录。测试团队需在集成测试阶段执行"跨用户数据隔离"专项测试用例。

### 2.2 搜索入口交互设计

| 交互元素 | 规格说明 |
|---------|---------|
| 搜索框 placeholder | 根据入口不同展示差异化文案：爱问搜索显示"搜索我的对话"，资源搜索显示"搜索资源标题/关键词"，社区搜索显示"搜索话题标题/内容" |
| 搜索面板 | 点击搜索框展开全宽搜索面板，支持即时输入 |
| 搜索触发 | 回车键或点击搜索按钮触发搜索 |
| 搜索取消 | 点击取消按钮或点击搜索框外部收起面板 |

### 2.3 搜索索引范围与状态过滤规则

> **【核心安全规则】** 本节定义搜索索引的可见性边界，确保搜索结果不泄露非公开态内容。所有搜索请求在后端检索时必须携带状态过滤条件，未通过状态校验的内容不得出现在搜索结果中。

| 搜索入口 | 可索引内容状态 | 必须排除的内容状态 |
|---------|--------------|-----------------|
| 资源搜索 | `status = 已通过` | 待审核、被拒绝、申诉中、已下架；已封禁用户发布的资源 |
| 资源搜索补充说明 | **含精华标记资源**：管理员加精后状态不变（仍为 已通过），搜索索引可见，排序规则详见 §4.2（MVP阶段加精不影响搜索权重） | — |
| 社区搜索 | `status IN (待回答, 讨论中, 已解决, 沉寂)` 且 `deleted_at = NULL` 且发布者未被封禁 | 待审核（如有）、已删除、举报待审、已封禁用户的话题 |
| 社区搜索补充说明 | **含精华标记话题**：管理员加精后话题状态不变（仍为 待回答 / 讨论中 / 已解决 其中之一），搜索索引可见，排序时叠加 `Featured_Boost`（详见 §4.3） | — |
| 爱问搜索 | `user_id = 当前登录用户 UID` | 其他用户的对话记录（由 §2.1 数据隔离规则强制执行） |

**后端实现要求**：每次搜索查询必须将状态过滤条件作为 WHERE 子句的必选条件注入，禁止绕过。搜索结果返回前，服务端须对结果集执行**二次状态校验**（Re-validation），过滤掉在索引同步窗口期内状态已变更（如刚被删除/下架）的"幽灵结果"，确保用户点击后不会遇到 404。

## 三、搜索算法与检索策略

### 3.1 PostgreSQL 全文检索实现

MVP 阶段采用 PostgreSQL 内置全文检索能力（tsvector / tsquery），实现以下检索功能：

| 检索类型 | 实现方式 | 说明 |
|---------|---------|------|
| 关键字精准匹配 | 分词（to_tsquery）精确检索 | 完整匹配用户输入的关键词 |
| 关键字模糊匹配 | 分词 + 前缀/后缀模糊 | 支持 `*` 前缀/后缀通配符匹配 |
| 多关键字组合 | 布尔检索（AND / OR） | 空格分隔多关键字，默认 AND 关系 |

**检索模式选择规则**：
- 当用户输入不含通配符的完整词语时，使用精准检索模式
- 当用户输入包含 `*` 通配符时，切换为模糊检索模式
- 当用户输入多个空格分隔的词语时，使用布尔 AND 组合检索

> **【SQL 注入防护 — 强制约束】** 搜索词必须通过 `plainto_tsquery()` 或**参数化查询（Bind Parameter / Prepared Statement）** 方式传入 SQL，**严禁字符串拼接**。所有用户输入中的特殊字符（`'`, `"`, `;`, `--`, `\`）必须在服务端进行转义或由 ORM 框架自动处理。违反此规则的代码不得通过 Code Review。

### 3.2 分词策略

| 分词组件 | 技术选型 | 说明 |
|---------|---------|------|
| 中文分词器 | jieba（python-jieba 或 pg_jieba 扩展） | 支持中文按词分词，支持用户自定义词典 |
| Ngram 分词 fallback | pg_jieba 支持的字符级别 ngram | 当按词分词无法覆盖时，使用字符级 ngram 补充 |
| 自定义词库 | 可靠性领域专业术语库 | 包含可靠性工程国家标准术语、电子元器件术语、测试方法术语等 |

**自定义词库管理**：
- 自定义词库（SUP-CFG-WORDLIB）的管理入口对应管理后台 PRD §5.9「敏感词库管理」界面，由超级管理员授权后操作
- 当前 MVP 阶段复用管理后台 §5.9 敏感词库管理界面实现词的增删改查，待 Phase 2 扩展为独立的"搜索分词词库"管理功能
- 词库变更后需主动触发搜索索引的增量更新（由 Phase 2 搜索升级实现，当前 MVP 阶段通过定时全量重建索引覆盖）

### 3.3 搜索命中权重

> **【适用范围说明】** 本节定义的权重系数**仅适用于资源搜索和社区搜索**。爱问搜索以对话发起时间倒序为主要排序依据，不适用以下权重体系。

检索结果 relevance score 计算规则如下：

| 命中字段 | 权重系数 | 说明 |
|---------|---------|------|
| 标题（title）命中 | **3.0** | 标题完全匹配或分词命中 |
| 简介/内容（summary / content）命中 | **1.5** | 简介或正文内容命中 |
| 关键词（tags / keywords）命中 | **2.0** | 用户填写的标签或关键词命中 |
| 多字段同时命中 | 叠加计算 | 多字段同时命中的 score 相加 |

Relevance Score 计算公式：
```
relevance_score = title_weight * title_hits + summary_weight * summary_hits + keyword_weight * keyword_hits
```
其中 title_weight = 3.0，summary_weight = 1.5，keyword_weight = 2.0。

## 四、排序规则与热度权重

### 4.1 爱问搜索排序规则

| 排序维度 | 说明 |
|---------|------|
| **默认排序** | 按**对话最后活动时间**倒序排列（最新靠前） |
| 时间倒序 | 最新对话（最后回复时间）排在最前 |
| 时间正序 | 最老对话排在最前 |

爱问搜索不采用相关性 score 排序，原因：用户历史对话数量有限（通常数十至数百条），时间维度是用户定位目标对话的最高效维度。

> **【排序口径对齐审计报】** 本节排序规则与 PRD_MVP_爱问模块对齐：以"对话**最后活动时间 (updated_at)**"为默认排序字段。确保用户最近聊过的会话始终出现在最上方。

> **爱问搜索性能基线**：对齐 PRD_MVP_爱问模块 M1-F016，以"用户实际历史记录规模（如 30-100 条）"为测试场景，搜索结果返回 <= 1 秒。

### 4.2 资源搜索排序规则

**默认排序**：按**相关性（relevance_score）** 优先，其次按**综合热度指数（Heat_Index）** 降序排列。

**热度指数计算公式**（基础公式对齐 PRD_MVP_资源模块 §4.2，搜索系统叠加时间衰减修饰）：
```
Heat_Index = MAX(0, [(download_count * 10) + (like_count * 5) - (dislike_count * 8) + (comment_count * 3) + (distinct_views * 1)]) * (1 / (T_diff + 2)^1.5)
```
- `T_diff`：资源上线至今的天数偏差，公式中 +2 是为避免 T_diff=0 时分母为 1。
- 采用幂律衰减（Power Law Decay）实现自然老化，替代离散的二元 Aging_Factor。
- 详见《集成_全文检索算法方案.md》§5.1。

> **【非负底线约束】** 热度指数经 `MAX(0, ...)` 钳位处理，确保不出现负值。当点踩量远超正向指标时，热度归零而非变为负数，避免排序逻辑异常。

**【标签与排序关系】**：
- "最新（7天）"、"精华"、"基础资源"、"闲置"等标签属于**展示标签**（定义详见《PRD_内容生命周期管理.md》§2.3），用于列表筛选，不直接影响搜索排序。
- 搜索排序由幂律衰减公式自动实现自然老化：超过 7 天的资源热度会自然衰减至约 **3.7%**，超过 30 天衰减至约 **1.5%**。
- 管理员可通过"精华"标签在 Phase 2 引入 `Featured_Boost` 加权。

| 热度因子 | 系数 | 说明 |
|---------|------|------|
| 下载量（download_count） | x10 | 资源被下载次数，正向热度 |
| 点赞量（like_count） | x5 | 资源点赞次数，正向热度 |
| 点踩量（dislike_count） | x8 | 资源点踩次数，负向热度（点踩越多，热度越低） |
| 评论互动量（comment_count） | x3 | 资源评论次数，正向热度（对齐资源模块 PRD §4.2） |
| 去重浏览量（distinct_views） | x1 | UV 去重浏览量（基于用户 IP + 设备指纹去重，24 小时内同一用户对同一资源的重复访问仅计 1 次），正向热度 |

> **【资源精华标签与搜索排序】** 资源"精华"标签（由管理员标记，定义见 PRD_内容生命周期管理 §2.3）在 MVP 阶段**仅用于列表展示和分类筛选，不影响搜索排序权重**。Phase 2 可评估是否为资源引入 `Featured_Boost` 加权。

**支持排序维度**：相关性（默认）、综合热度（下载量+点赞）、最新上架（按 created_at 倒序）。

### 4.3 社区搜索排序规则

**默认排序**：按**相关性（relevance_score）** 优先，其次按**讨论热度（Heat_Index）** 降序排列。

**热度指数计算公式**（基础公式对齐 PRD_MVP_社区模块 §3.2，搜索系统叠加时间衰减修饰）：
```
Heat_Index = MAX(0, [(reply_count * 5) + (like_count * 3) - (dislike_count * 5) + (distinct_views * 0.5)]) * (1 / (T_diff + 2)^1.5) + Pinned_Boost + Featured_Boost
```
- `T_diff`：话题发布至今的天数偏差，公式中 +2 是为避免 T_diff=0 时分母为 1。
- 采用幂律衰减（Power Law Decay）实现自然老化，替代离散的二元 Aging_Factor。
- 详见《集成_全文检索算法方案.md》§5.1。

> **【非负底线约束】** 与资源热度一致，公式主体部分经 `MAX(0, ...)` 钳位后再叠加 Pinned_Boost 和 Featured_Boost，确保加权值不会被负热度抵消。

| 热度因子 | 系数 | 说明 |
|---------|------|------|
| 回复数（reply_count） | x5 | 话题回复讨论次数，正向热度 |
| 点赞量（like_count） | x3 | 话题点赞次数，正向热度 |
| 点踩量（dislike_count） | x5 | 话题点踩次数，负向热度 |
| 去重浏览量（distinct_views） | x0.5 | UV 去重浏览量（基于用户 IP + 设备指纹去重，24 小时内同一用户对同一话题的重复访问仅计 1 次），正向热度 |

**置顶加权系数（Pinned_Boost）**：

| 话题状态 | Pinned_Boost 值 | 说明 |
|---------|---------------|------|
| 非置顶话题 | **0** | 不享受置顶加权 |
| 管理员手动置顶话题 | **100** | 固定加分值，确保在关键词匹配前提下始终排列于搜索结果第一屏 |

> **【MVP 配置说明】** Pinned_Boost 当前为**硬编码固定值**（100），MVP 阶段不提供管理后台动态配置入口。如需调整，由研发团队修改配置常量后重新部署。Phase 2 将在管理后台建设独立的"搜索排序配置"功能模块后提供在线配置能力。

**精华加权系数（Featured_Boost）**：

| 话题状态 | Featured_Boost 值 | 说明 |
|---------|-----------------|------|
| 非精华话题 | **0** | 不享受精华加权 |
| 管理员标记精华话题 | **50** | 固定加分值，精华话题在同等相关性条件下优先展示，但优先级低于置顶 |

> **【MVP 配置说明】** Featured_Boost 当前为**硬编码固定值**（50），MVP 阶段不提供管理后台动态配置入口。精华标记功能定义见 PRD_MVP_社区模块 §3.4。Phase 2 将在管理后台建设独立的"搜索排序配置"功能模块和社区模块 PRD 补充精华加权参数规范后提供在线配置能力。

**支持排序维度**：相关性（默认）、讨论热度（回复数+点赞）、最新发布（按 created_at 倒序）。

## 五、用户体验优化

### 5.1 联想词提示

| 规格项 | 详细说明 |
|-------|---------|
| 触发机制 | 用户输入关键字>=1 个字符后，采用 **200ms 防抖（Debounce）** 触发联想词查询。即用户停止输入 200ms 后才触发一次查询；若 200ms 内用户继续输入则重新计时，确保只有最终输入触发查询 |

| 联想词来源 | 平台近 30 天高频搜索词（去重后） + 可靠性专业术语词库 |

## 七、搜索日志与审计规范

### 7.1 日志记录项

每次搜索请求完成后，需在服务侧落库以下日志字段：

| 字段名 | 数据类型 | 说明 |
|-------|---------|------|
| search_id | UUID | 本次搜索唯一标识 |
| user_id | UUID | 搜索用户 UID（游客为 NULL） |
| device_fingerprint | VARCHAR(64) | 游客设备指纹 |
| search_keyword_hash | VARCHAR(128) | 搜索词 SHA-256 哈希（**合规审计表**） |
| search_entry | ENUM | 搜索入口：`AIQA` / `RESOURCE` / `COMMUNITY` |
| hit_count | INT | 命中结果数量 |
| response_time_ms | INT | 服务端响应耗时（毫秒） |
| filter_applied | JSONB | 应用的筛选/排序条件 |
| timestamp | DATETIME | 搜索发生时间（UTC+8） |

### 7.2 联想词统计表（独立）

> **【合规拆分说明】** 为解决"搜索词脱敏"与"高频词统计"的需求冲突，系统拆分为两张独立表：

| 表名 | 存储内容 | 用途 |
|------|---------|------|
| `search_audit_log` | search_keyword_hash + 完整审计字段 | 审计追溯、异常检测、合规 |
| `search_keyword_stats` | 脱敏后搜索词明文 + 频次统计 | 联想词生成、热门词排行 |

- `search_keyword_stats` 中的搜索词在写入前须经**敏感词库过滤**，命中黑名单的词汇不录入。
- 两张表通过 `search_id` 关联，但访问权限隔离：审计表仅管理员可查，统计表供搜索服务只读访问。

### 7.3 数据保留与合规

- **保留周期**：搜索日志保留 **180 天**，超期脱敏归档
- **日志隔离**：搜索日志与业务数据库隔离存储
- **合规说明**：审计表存储哈希值，统计表存储脱敏后明文（经敏感词过滤）

| 内容类型 | 索引时机 | 最大同步延迟（工程目标值，待技术方案验证后确认） | 索引失败处理 |

| 资源审核通过 | 审批通过操作完成时 | **<= 5 秒**（目标值） | 写入失败队列，定时任务重试（最多 3 次）；3 次均失败后触发 **M5 管理后台告警**，由运维手动介入 |
| 话题发布 | 提交发布完成时 | **<= 5 秒**（目标值） | 同上 |
| 资源/话题状态变更 | 状态变更完成时 | **<= 5 秒**（目标值） | 同步删除/更新索引状态字段 |
| 热度分数更新 | 定时任务**每小时**执行 | <= 1 小时 | 增量更新热度分数 |

> **【Phase 2 热点探测预留】** 当单个资源/话题在 1 小时内交互增量超过阈值（如 100 次），可触发即时热度更新，缩短热度刷新延迟。此能力归入 Phase 2 优化。

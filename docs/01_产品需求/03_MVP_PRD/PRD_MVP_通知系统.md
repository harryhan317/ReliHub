## 一、 模块目的与范围边界

### 1.1 文档目的
本模块（SUP-F011~SUP-F019）旨在定义 ReliHub 平台的通知系统规范。通过建立统一的消息触达中心，确保系统告警、业务动态、社交互动能准确、实时地送达用户，从而提升用户参与度并保证业务流程（如审核、反馈）的逻辑闭环。

### 1.2 核心范围 (对齐功能清单 §9.2)
- **站内信系统 (In-app Message)**：持久化的消息中心，支持分类浏览。
- **实时触达 (Real-time Delivery)**：基于 WebSocket 或长轮询机制的红点提醒与顶部横幅。
- **消息生命周期**：涵盖消息的触发、网关路由分发、阅读状态切换及归档/匿名化。

### 1.3 边界说明 (MVP vs Phase 2)
- **站内实时推送 (MVP)**：指用户在线时，系统通过 WebSocket 协议下发的即时反馈（弹窗/红点）。此类通道在 MVP 阶段即需实现。
- **三方系统级 Push (Phase 2)**：指手机 OS 厂商提供的离线推送服务（APNs, FCM 等）。MVP 阶段暂不接入。
- **短信通知范围**：除核心安全告警和验证码外，日常生产业务通知暂不接入短信提示。

## 二、 业务架构与管控模型

### 2.1 消息触达链路 (Architecture)
1. **触发层 (Trigger)**：业务模块（M1-M5）产生特定事件（如：评论被回复、资源审核通过、AI 对话限额报警）。
2. **加工与路由网关层 (Processing & Routing Gateway)**：进行消息模板匹配、参数注入（如用户名、单号）、消息聚合判定及跳转 Scheme 生成。
3. **分发层 (Dispatcher)**：处理消息的原子性入库，并触发实时 UI 推送信号。针对管理后台广播消息执行异步分批削峰下发。
4. **呈现层 (Presentation)**：用户端红点、列表更新、应用内 Banner 提醒。

### 2.2 幂等性与防骚扰规则
- **去重逻辑**：同一触发事件在 5s 内不得产生重复通知。
- **聚合机制**：互动类消息（如点赞、收藏）支持“XX 等 N 人”的聚合逻辑，避免消息列表刷屏。
    - **逻辑说明**：系统基于 `category + reference_id + date` 生成 `aggregate_key`，24 小时内的同类互动在列表页仅展示一条聚合记录，点击进入详情可见完整列表。

---

### 3.1 消息中心入口与布局 (SUP-F014/F016)
- **物理入口**：
    - **主入口**：APP 顶端导航栏（Top Bar）右侧的 **“铃铛”图标**，点击进入通知中心。
    - **次入口**：位于“我的”模块功能列表中。
- **视觉显示**：图标右上角展示**全局未读数 (Global Badge)**。
- **三个分类 Tab**：
    - **系统通知 (System)**：规则更新、账号安全、处罚等。
    - **互动通知 (Interaction)**：点赞、回复、采纳、@提醒。（对齐《术语表》§7-124）
    - **服务反馈 (Service)**：资源审核结果、异议回复、工单结单。
- **一键已读 (Mark All as Read)**：分类列表底部或右上角支持“本类已读”及“全局已读”功能。点击后，所有未读消息记录的 `is_read` 状态批量更新为 `true`，对应红点消失。

### 3.2 提醒交互 (SUP-F015/F017)
- **未读红点**：Tab 级累加数字，进入列表后按分类清除。
- **全局未读数 (Global Badge)**：导航栏或主入口显示的数字，等于三个子分类未读数之和（`SUM(unread_count)`）。
- **全局拦截弹窗 (WebSocket Push)**：针对高优先级消息（如封号、强退、反馈结单）在用户在线时即时弹出横幅。

### 3.3 深度跳转 (Deep Linking) (SUP-F019)
- 所有通知项必须支持一键跳转 (Deep Linking)。点击通知后，系统依据 `jump_url` 字段自动定位到目标业务页。
- ** Scheme 协议规范**：
    - **资源详情**：`relihub://resource/detail?id={resource_id}`
    - **话题详情**：`relihub://community/topic?id={topic_id}`
    - **反馈详情**：`relihub://feedback/detail?id={ticket_id}`
    - **AI 对话页**：`relihub://ai/chat?id={session_id}`
    - **申诉详情**：`relihub://appeal/detail?id={appeal_id}`（对应 SUP-F013 申诉回复通知）
    - **安全中心**：`relihub://user/security`
    - **可可豆明细**：`relihub://user/assets/beans`

### 3.4 用户通知设置 (SUP-F018)
- **定义**：用户可自主管控各类消息的接收开关。
- **控制粒度**：
    - **按类型开关**：系统、互动、服务。
    - **按通道开关**：站内消息列表（默认置顶且不可关闭）、实时推送（WebSocket 弹窗可关闭）。
- **免打扰时段 (DND)**：支持设置时间段（如 23:00~08:00）自动静默实时推送信号（无横幅弹窗），但消息仍正常入收件箱，用户主动刷新时可见未读状态。
    - **技术方案**：WebSocket 推送信号包含 `is_dnd_active` 状态位或 `priority` 标识，前端依据此状态及用户本地设置决定是否弹出 Banner。

## 四、 功能详解：管理侧能力 (M5 对接)

### 4.1 广播与定向推送 (M5-F011)
- **站内信群发**：超级管理员具权向全量用户推送系统公告。
- **模板化配置**：支持在后台定义消息文案变量。
- **防骚扰约束**：全量广播消息每自然日上限 **1 次**（防止运营滥用），每次群发需填写操作原因并记录审计日志。
- **免打扰时段豁免**：广播类系统公告属于高优先级触达，不受用户 DND 免打扰时段约束（这是行业惯例，确保重大规则变更、安全告警可全量送达）。
- **广播消息优先级**：字段 `priority` = HIGH（高优先级），走独立推送通道，不受用户通知类型开关约束。

## 五、 消息分类与模板规范

### 5.1 系统与安全类
- **账号异常**：`【安全提醒】您的账号在异地登录（IP: {ip}），如非本人操作请立即修改密码。`
- **等级变动**：`【等级升级】恭喜！您的信誉分已达标，现已晋升为{level_name}。`
- **AI 额度预警**：`【额度提醒】您的本月 AI 对话可用额度已不足 20%，请注意合理规划使用。`

### 5.2 互动与奖励类
- **获得互动**：`{user_name} 评论了您的资源《{resource_title}》。`
- **采纳通知**：`【悬赏达成】您在话题中的回答已被采纳，奖励 {amount} 可可豆已发放到账。`
- **邀请奖励**：`【邀请成功】您邀请的好友已完成注册，{amount} 可可豆奖励已发放。`

### 5.3 服务/审核类
- **审核结果**：`【审核通过】您的资源《{resource_title}》已发布。`
- **悬赏过期退款**：`【退款通知】您的悬赏话题因 30 天未采纳最佳回答，悬赏金 {amount} 可可豆已原路退回。`
- **反馈结单 (对齐意见反馈V1.5)**：`【反馈处理结果】您的反馈（单号：{ticket_id}）已有处理结果。管理员回复：{reply_content}。点击查看详情。`

## 六、 性能与非功能需求 (NFR)

### 6.1 实时性与并发 (SLA)
- **触达延迟**：业务触发到消息落库并推送红点 ≤ 1秒 (对齐 NFR §2.2.2)。
- **查询 SLA**：消息列表加载时间 ≤ 500ms（单页 100 条数据）。

### 6.2 数据留存
- **消息数据作为审计链路的一部分，永久留存**（对齐《PRD_产品功能清单.md》SUP-F016 及意见反馈 PRD 数据存留策略）。用户执行"删除"时为**逻辑删除**（用户端不可见），后台审计数据仍永久保留。

## 七、 数据库字段预定义 (Appendix A)

| 字段名 | 类型 | 说明 |
|-------|------|------|
| `msg_id` | UUID | 消息唯一 ID (PK) |
| `receiver_uid` | UUID | 接收人 ID |
| `sender_id` | UUID | 发送方 ID (系统机器人为 0 补位的 UUID，管理员为管理员 UID) |
| `sender_type` | TINYINT | 1:系统, 2:管理员, 3:业务流 |
| `category` | TINYINT | 分类 (1:SYSTEM, 2:INTERACTION, 3:SERVICE, 4:AUDIT, 5:BROADCAST) |
| `priority` | TINYINT | 优先级 (1:普通-NORMAL, 2:高优先级-HIGH) |
| `ref_type` | VARCHAR | 关联业务类型 (如: RESOURCE, POST, FEEDBACK) |
| `ref_id` | UUID | 关联业务主键 ID |
| `aggregate_key` | VARCHAR | 用于聚合逻辑的 Hash Key (如类型+业务ID+日期) |
| `content_json` | JSONB | 结构化内容及变量数据 (包含 reply_content 等摘要) |
| `jump_url` | VARCHAR | 跳转 Scheme 协议 |
| `is_read` | BOOLEAN | 是否已读 |
| `is_deleted` | BOOLEAN | 逻辑删除标记（用户端不可见） |
| `created_at` | TIMESTAMP | 生成时间 |

## 八、 验收标准 (Acceptance Criteria)
1. 验证三种分类下的消息能否准确依据业务触发下发。
2. 验证红点及实时弹窗（WebSocket）在设置关闭后是否按预期静默。
3. 验证点击消息后的深度跳转是否准确到达对应详情页。
4. 验证用户注销后，对应消息记录是否执行匿名化且留存。
5. 验证 SUP-F018 通知开关：关闭互动类通知后，点赞/回复等互动消息不再出现在消息列表及红点中；开启后恢复正常。
6. **性能压测验收**：验证在 100 万级广播下发时，消息列表的分页查询（OFFSET 10000+）耗时仍控制在 1s 以内。
7. **一致性验收**：验证互动消息聚合后，进入详情页能看到该 `aggregate_key` 下的所有原始交互明细。

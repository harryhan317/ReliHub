## 一、 模块目的与范围边界

### 1.1 文档目的
本模块（SUP-F046~F051）旨在建立 ReliHub 平台用户与运营团队之间的非实时异步沟通渠道。通过规范化的反馈采集与透明的工单闭环流程（由“意见反馈”工单承载），及时发现系统漏洞、收集产品建议并辅助纠正社区内容偏差，确保电子产品可靠性领域专业平台的质量稳步提升。

### 1.2 涵盖范围与优先级 (对齐功能清单 §9.6)
本模块功能定义严格遵循《PRD_产品功能清单.md》基准，在术语表 (§7-125) 中统称为“意见反馈”：

| 功能编号 | 功能名称 | 优先级 | 核心需求描述 |
|---------|---------|-------|-------------|
| SUP-F046 | 表单反馈 | S3 | 支持文字描述及多图附件上传 |
| SUP-F047 | 联系方式展示 | S3 | 展示官方微信/邮箱，引导紧急问题转线下 |
| SUP-F048 | 反馈单号生成 | S3 | 自动生成唯一可溯源的工单编号 |
| SUP-F049 | 反馈处理流程 | S3 | 后台工单流转逻辑（待处理→处理中→已解决/驳回） |
| SUP-F050 | 反馈结果通知 | S3 | 自动化站内消息反馈结单结果 |
| SUP-F051 | 满意度评价 | S3 | 用户对已解决反馈进行评价 |

### 1.3 不涵盖范围
*   **反馈奖励激励 (SUP-F057)**：划归 Phase 2，后期视运营需要接入可可豆奖励。
*   **IM 即时通讯**：MVP 阶段仅支持工单回帖式异步互动。

## 二、 核心业务逻辑与时序管控

### 2.1 工单状态流转规则
反馈工单生命周期由以下状态机驱动，严禁逆向流转：

1.  **待处理 (Pending)**：
    *   **触发条件**：用户成功提交表单。
    *   **可见性**：对所有具反馈管理权限的管理员可见于"公共领用池"。
2.  **处理中 (Processing)**：
    *   **触发条件**：管理员执行"领用/抢单"动作。
    *   **锁定机制**：一旦进入此状态，该工单与该管理员建立唯一绑定，其他管理员不可再行处置。
3.  **已解决 (Resolved)**：
    *   **触发条件**：管理员提交处理报告并选择"确认解决"。
    *   **交互约束**：管理员执行此操作时，**必须填写**"对外回复内容" (`reply_content`)，以向用户明确处理结论。
3a. **已驳回 (Rejected)**：
    *   **触发条件**：管理员认定为恶意刷屏、内容重复或超出服务范畴。
    *   **交互约束**：管理员执行此操作时，**必须填写**"对外回复内容" (`reply_content`)，以向用户明确驳回原因。
4.  **例外规则：超级管理员权限**：
    *   **强行释放/转办**：针对长期处于 [Processing] 但处理人离职或无响应的"孤儿工单"，超级管理员具权强行将其状态重置回 [Pending] 或直接指派给其他管理员。

### 2.2 双层 SLA 约束对齐
*   **接口响应 SLA**：
    *   **表单提交 (写入类)**：<= 1 秒（对齐《PRD_非功能需求.md》§2.2.2 写入类接口标准）。
    *   **反馈历史加载 (查询类)**：<= 500ms（对齐《PRD_非功能需求.md》§2.2.2 查询类接口标准）。
*   **业务处理 BLA**：
    *   **48 小时结单承诺**：从[待处理]变更为[已解决/已驳回]的物理耗时不得超过 48 小时（对齐《PRD_MVP_管理后台.md》§7.5）。
    *   **超时预警**：超过 48 小时未结单，后台列表触发高亮置顶预警。

## 三、 功能详解：用户侧体验

### 3.1 提交入口与表单 (SUP-F046/F047)
*   **入口路径**：
    1.  **正常入口**：【我的】→【设置】→【意见反馈】。
    2.  **故障辅助入口**：在【登录/注册页】显著位置展示“遇到问题？联系客服”（对应 SUP-F047），展示官方微信号/邮箱，确保无法登录的用户亦能通过非表单渠道反馈。
*   **防刷限制**：同一 UID 每日提交上限为 **5 条**，超出时提示“反馈过于频繁，请明日再试”。
*   **表单字段规范**：
    *   **反馈类型**：功能异常、体验建议、内容纠错、其他（单选）。
    *   **反馈内容**：必填，10~1000 字符。
    *   **附件图片**：最多 3 张；支持 `png, jpg, jpeg, gif, bmp, webp, tif, tiff`；单文件 <= 5MB（对齐《PRD_文件与附件处理规范.md》§2.1）。所有附件需经过 **AI 安全预审 + 病毒扫描**。
    *   **联系方式**：选填，字段名 `contact_info`。
*   **上下文关联机制**：
    *   若用户从【资源详情页】或【社区话题页】发起反馈，系统需自动捕获 `reference_id` (资源ID/话题ID) 及 `reference_type`，以便管理员精准定位内容纠错对象。
*   **工单历史**：用户可查看历史工单列表与进度，分页规格为 **20 条/页**。

### 3.2 满意度评价 (SUP-F051)
*   **触发时机**：工单变为"已解决"后，用户可点击"评价"进行打分。
*   **评价窗口期**：结单后 **7 个自然日**。逾期未评价则该工单自动转为“系统默认满分评价”（不落 evaluation_body），评价入口关闭。
*   **MVP 评价框架**：满意/不满意 + 可选文本（<=100字），评价结果落库供查询。
*   **Phase 2 演进细化项**：评价结果统计报表、客服质量考核联动、评价激励（SUP-F057）。

## 四、 功能详解：管理侧处理 (M5 对接)

### 4.1 抢单与协作机制 (SUP-F049)
*   **领用模式 (Claim)**：采用公共池领用制。管理员点击处理后，系统将 `handler_id` 写入记录。
*   **回复限制**：管理员回复内容 (`reply_content`) 长度上限为 **2000 字符**。
*   **重复识别**：系统基于近期（24h内）同一用户的反馈内容进行 MD5/语义相似度辅助检测，提示管理员是否存在重复投诉。
*   **分页规格**：管理后台反馈列表每页展示 **50 条**。

### 4.2 结果通知闭环 (SUP-F050)
*   **通知原子性**：通知下发为结单动作的**同步伴随步骤**——结单事务提交成功即触发通知下发，不存在独立的"通知超时"打断结单流程。
*   **失败容忍**：若通知下发失败（如用户未激活站内信账户），系统记录失败日志（不重试，不阻塞结单）；通知重试逻辑由站内信模块定义（详见站内信 PRD）。
*   **通知内容规范**：
    *   **模板**：`【反馈处理结果】您的反馈（单号：{ticket_id}）已有处理结果。管理员回复：{reply_content}。点击查看详情。`
    *   通知必须包含管理员的回复摘要，确保用户无需跳转即可获知核心结论。

## 五、 数据库字段命名规范 (Appendix A)

| 字段名 (snake_case) | 数据类型 | 说明 | 必填 |
|--------------------|---------|-----|------|
| `ticket_id` | VARCHAR | 工单**业务单号**。格式：`FB + YYYYMMDD + 6位流水` | 是 |
| `user_id` | UUID | 提交人唯一标识 (FK -> users.id) | 是 |
| `feedback_type` | TINYINT | 类型枚举 (1:异常, 2:建议, 3:纠错, 4:其他) | 是 |
| `reference_id` | UUID | 关联的内容 ID (资源 ID/话题 ID/会话 ID) | 否 |
| `reference_type` | TINYINT | 关联类型 (1:资源, 2:话题, 3:会话) | 否 |
| `content_body` | TEXT | 意见反馈正文 | 是 |
| `contact_info` | VARCHAR | 联系方式（手机/邮箱）。**普通管理侧展示需脱敏**；**超级管理员可见原文**以备紧急线下联系。 | 否 |
| `attachments_urls` | JSONB | 图片附件 URL 数组 | 否 |
| `status` | VARCHAR | 状态 (PENDING, PROCESSING, RESOLVED, REJECTED) | 是 |
| `handler_id` | UUID | 处理管理员 UID (FK -> users.id) | 否 |
| `reply_content` | TEXT | 管理员回复。状态为 REJECTED 时必填；长度上限 2000 字符 | 否 |
| `internal_notes` | TEXT | 管理员内部备注（用户不可见） | 否 |
| `evaluation_score` | TINYINT | 满意度分值 (1:满意, 2:不满意) | 否 |
| `evaluation_body` | VARCHAR | 满意度评价文本（<=100字） | 否 |
| `evaluation_at` | TIMESTAMP | 用户评价时间 | 否 |
| `created_at` | TIMESTAMP | 提交时间 | 是 |
| `resolved_at` | TIMESTAMP | 结单关单时间 | 否 |

**特殊业务规则说明**：
1.  **用户注销/匿名化 (含 2 年被动注销)**：若提交人执行了账户注销或触发“2 年未登录被动注销”，出于审计需要，对应工单记录**严禁物理删除**。系统需：
    *   将 `user_id` 重置为匿名 UUID (例: `00000000-0000-0000-0000-000000000000`)。
    *   **保留 `anonymized_user_hash` (不含 PII 但唯一)**：通过 `HMAC-SHA256(original_user_id, system_salt)` 生成，以便管理员在后台仍能识别反馈项目的“聚集性”（判断是否来自同一注销实体）。
    *   物理清除 `contact_info` 明文原文，文件附件按生命周期策略物理删除，但工单文本明细永久存留。
2.  **附件安全态联动**：若反馈附件处于 `SCANNING` 或 `ISOLATED` 态（详见文件规范 §5.1），管理后台需在工单详情中显著标识“附件安全检测中”且拦截“确认解决”操作，直至检测完成。
3.  **数据存留策略**：所有反馈工单及评价数据作为系统审计日志的一部分，**永久存留**。

## 六、 测试验收与 NFR 指标

### 6.1 核心路径验证
| 场景编号 | 场景描述 | 预期结果 |
|---------|---------|---------|
| TC-FB-001 | 游客提交反馈 | 系统提示“请先登录”，阻断提交。 |
| TC-FB-002 | 正常提交反馈 | 生成合法 FB 单号，分页列表可见。 |
| TC-FB-002-C | 上下文提交 | 从资源详情页发起纠错，后端成功记录 `reference_id`。 |
| TC-FB-003 | 管理员抢单 | A 管理员领用后，B 管理员无法接单且状态同步为 PROCESSING。 |
| TC-FB-004 | SLA 超时检测 | 人工构造超 48h 记录，验证后台列表是否高亮预警。 |
| TC-FB-005 | 满意度评价 | 已解决工单用户可成功评价，评价状态同步落库。 |
| TC-FB-006 | 注销一致性 | 用户注销后，验证其历史工单 `user_id` 自动匿名化且记录存续。 |

### 6.2 性能指标
*   **反馈表单提交**：目标值 <= 1s。
*   **后端反馈列表加载**：目标值 <= 500ms（对齐 NFR §2.2.2）。
*   **附件图片上传**： <= 3s (5MB 典型环境)。

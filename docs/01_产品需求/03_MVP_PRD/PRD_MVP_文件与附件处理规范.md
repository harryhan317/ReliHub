## 一、文档目的与范围边界

### 1.1 文档目的
定义 ReliHub 平台各模块（AI 对话、资源中心、管理后台等）在文件上传、存储、安全校验、分发下发、预览及清理全链路中的技术基准。本规范解决跨模块调用的一致性，确保文件资产的安全、合规与低冗余存储。

---

## 二、文件格式与深度校验

### 2.1 格式白名单分类与 Magic Number

| 类别 | 允许后缀 | 校验方式 |
|------|----------|----------|
| **文档类** | pdf, doc, docx, txt, xls, xlsx, ppt, pptx | 文件头魔数 (e.g. `%PDF-` ) |
| **图片类** | png, jpg, jpeg, gif, bmp, webp, tif, tiff | 二进制流识别 + MIME Type |

### 2.2 格式处理细节
- **TIF / TIFF 预览支持**：由于浏览器原生不支持 TIF 格式直接渲染，此类文件在预览流水线中必须执行**强制转码**（转为 JPG/PNG），其逻辑同 §6.1 的异步转换流程。**转码完成后，预览流 SLA 重新从转码完成时开始计算，不继承原文件上传时间**，即 TIF 预览整体耗时可能超过 5s（转码耗时 + 水印合成耗时），其 SLA 指标另计。
- **ZIP 深嗅 (Sniffing)**：针对 `.docx / .xlsx / .pptx` （系统识别为 ZIP 格式），后端必须检测内部目录结构（如是否存在 `word/document.xml`）。

## 三、文件配额与限制阈值 (Quota Alignment)

### 3.1 单文件大小系统硬上限
- **文档/图片类**：**20 MB** (系统全局硬上限)；管理后台初始默认值为 **5 MB**（普通用户初始体验值）。

### 3.2 计费与配额归扣
1. **秒传扣减主体**：逻辑配额从**当前上传发起者**账号扣除。
2. **软删除状态消耗**：文件标记为 `DELETED` 但处于 30 天保留期内时，**依然占据用户的存储配额**。目的是防止恶意用户通过“删/填/删”循环无限占领平台存储资源池。
3. **物理强制粉碎 (Force Permanent Wipe)**：用户若需立即释放配额以继续上传，系统提供"物理强删"选项（操作不可逆，需二次验证确认）。执行后对当前用户产生以下原子性影响：① 执行者个人配额立即释放（不影响其他引用者）；② 执行者会话中该文件立即消失（`status` → `DELETED`，`lifecycle_status` → `PERMANENTLY_DELETED`）；③ 审计日志专项记录（不可删除/修改/覆盖）。OSS 物理删除则在全局 `ref_counts` 真正归零后由 Worker 异步执行（不超过 1 小时），不受任何单一用户的物理粉碎操作影响。

## 四、上传机制：秒传、分片与合并

### 4.1 全库秒传逻辑 (Instant Upload)
- **秒传判定**：上传前 MD5 匹配全库已有记录时，直接复用物理存储（节省 OSS 空间），逻辑配额从**当前上传发起者**账号扣除，`ref_counts` 全局引用计数 +1。
- **黑名单追溯拦截**：加入黑名单的 MD5 必须同步触发 **AI 对话附件内容解析 (M1-F011 多模态解析器)** 模块的实时校验。即使旧 Token 暂未失效，附件内容解析层在读取上下文时也应拦截涉敏 MD5。
- **状态不继承**：新引用的秒传文件须根据当前业务场景重新进行状态机流转。

### 4.2 分片上传与限流 (Multipart)
- **合并幂等 (Redis Lock)**：`/complete` 请求受分布式锁保护，有效期 300s，防止重试导致的元数据踩踏。
- **规模限制**：Max 20MB / 4 分片（MVP 规格）。

## 五、安全扫描与内容合规风控

### 5.1 异步扫描态排斥逻辑 (State Machine)
- **SCANNING 态**：全局排斥预览、下载、AI 解析。
- **隔离 (ISOLATED)**：30s 扫描超时后进入。触发 M5 业务报警，由管理后台人工复核确定准入或阻断。

### 5.2 内容审查流程 (AI 与 申诉)
- **实时拦截**：针对 AI 对话场景，敏感词命中后前端立即展示"文件涉敏，解读已阻断"红字提示。
- **申诉路径 (Appeal Path)**：用户对 AI 侧的文件拦截有异议时，点击"申请复核"按钮，该文件将以高优先级进入 **M5 管理后台 AI 可疑内容审核队列（见 PRD_MVP_管理后台 §6.2）**。管理员人工纠偏后，系统需将该 MD5 加入"白名单索引"。若该队列暂未覆盖 AI 附件场景，则由 M5 安全管理员在审计日志界面中手工处理，文件状态维护同步至 `file_meta.status`。

## 六、预览下发与防泄密安全

### 6.1 预览与转换流水线 (Word/PPT)
- **异步任务队列 (Task Queue)**：LibreOffice 转码属于 CPU/内存密集型任务，系统必须将其转入**独立任务队列**（如 Redis Task Queue）执行。
- **并发受限保护**：单应用节点限制并行转码任务数（如 ≤ 4），防止瞬时波动导致 OOM，超出负载时应排队处理。
- **指标说明**：5s 预览 SLA 仅覆盖水印流合成；转码本身独立计时。对于需要转码的文件（Word/PPT/TIF），系统必须在 **5s 内返回"转码中"状态码或 UI 信号（如 loading 动画）**，让用户感知请求已被接受并处理中；具体内容展示随转码完成异步下发，不纳入 5s SLA 考核。

### 6.2 明文水印与数字盲水印分级
1. **明文水印 (Visible Watermark)**：预览下行流中实时合成，覆盖所有用户。内容为"ReliHub+用户昵称+用户ID+时间戳"，浅色叠加不影响阅读，用于防止截图/录屏截取。
2. **数字盲水印 (Invisible Digital Watermark)**：仅限资源中心 (M2) 下载流执行。将用户名、时间戳等以频域/空域隐写算法嵌入文件内容，**人眼不可见，需专业工具提取**，用于泄露溯源追踪。为防止高并发下载压垮 CPU，应优先采用 **预生成缓存 (UID-based Cache)** 策略或限额异步植入。

## 七、数据架构与审计 (Infrastructure)

### 7.1 文件元数据透明化
- **引用透视**：管理后台显示某 FileHash 被哪些 UID、Conversation_ID 引用。

### 7.2 核心审计规范 (Hash Chaining)
- **高风险生命周期动作**：以下两个生命周期突变动作必须强制写入审计哈希链，防止管理员或黑客通过非法修改生命周期状态来恢复或销毁关键证据：
  - **RESTORE（恢复）**：`lifecycle_status` 从 `SOFT_DELETED` 恢复为 `ACTIVE`，触发审计记录（操作人/时间戳/恢复原因）。
  - **PERMANENTLY_WIPE（物理粉碎）**：`lifecycle_status` 被强制写入 `PERMANENTLY_DELETED`，触发审计记录（操作人/时间戳/触发来源：用户主动或 30 天过期）。
- **路径审计**：由于 `oss_path` 作为字段存储，任何导致存储路径变迁的操作（手动迁移/生命周期转冷）都必须强制写入**审计哈希链**，防止通过伪造路径指向恶意文件包。
- **审计动作枚举完整覆盖**：操作类型枚举必须包含但不限于：`UPLOAD`（上传）/ `DELETE`（软删除）/ `RESTORE`（恢复）/ `PERMANENTLY_WIPE`（物理粉碎）/ `PATH_CHANGE`（路径变更）/ `SCAN_RESULT_CHANGE`（扫描结果变更）/ `WHITELIST_ADD`（白名单加入）/ `BLACKLIST_ADD`（黑名单加入）。
- **对齐规范**：对齐 [PRD_MVP_管理后台 §8.4](./PRD_MVP_管理后台.md)。

## 八、文件生命周期与清理 (Lifecycle)

### 8.1 物理清理
- **碎片 (Fragments)**：24 小时内由 Job 物理清理。
- **游客 Session**：登出/过期后（24h）立即执行物理清理。
- **软删除数据**：30 天后，`lifecycle_status = SOFT_DELETED` 的文件由 Worker 同时执行 `lifecycle_status` → `PERMANENTLY_DELETED` 及 `status` → `DELETED`（符合双状态映射准则），并同步执行 OSS 删除；用户主动执行"物理强制粉碎"时同理，`lifecycle_status` 和 `status` 同步写入终结态，Worker 立即执行 OSS 删除（异步，不超过 1 小时），配额立即释放。

## 九、附录：核心数据结构 (file_meta) 
| 字段名 | 描述 | 索引建议 |
|--------|------|----------|
| `file_uuid` | 业务唯一标识 | PK |
| `file_hash` | 全量 MD5 值 | UNIQUE |
| `oss_path` | 存储绝对路径（变更需触发审计） | - |
| `ref_counts` | 逻辑引用计数 | - |
| `status` | NORMAL/SCANNING/ISOLATED/SUSPICIOUS/BLOCKED/DELETED | INDEX |
| `lifecycle_status` | ACTIVE / SOFT_DELETED / PERMANENTLY_DELETED，标识文件生命周期所处阶段。PERMANENTLY_DELETED 一旦写入不可逆转，所有生命周期清理 Worker 以此为终结信号 | - |
| `uploader_uid` | 归属用户 | INDEX |

> **双状态字段映射准则（强制约束）**：
> - 当 `lifecycle_status` 为 `SOFT_DELETED` 或 `PERMANENTLY_DELETED` 时，`status` 必须强制同步为 `DELETED`，不允许出现"删除态但 status 仍为 NORMAL"的逻辑冲突。
> - 当 `lifecycle_status` 为 `ACTIVE` 时，`status` 才允许在 `NORMAL / SCANNING / ISOLATED / SUSPICIOUS / BLOCKED` 之间正常流转。
> - 业务代码必须在所有修改 `lifecycle_status` 的代码路径上强制校验此映射，不允许绕过。

## 十、验收标准 (Acceptance Criteria)

1. **配额强制粉碎验证**：模拟配额已满场景，通过“物理强删”旧文件后再次上传，验证额度立即释放成功。
2. **AI 解析黑名单拦截**：黑名单录入后，已上传的附件在 AI 模块后续的“读取上下文”环节需返回空结果。
3. **并发转码抗压**：模拟 10 个以上并发转码请求，验证系统能够进入队列有序处理，且无 OOM 发生。
4. **TIF 预览转码**：上传 TIF 文件，验证前端预览展示正常的 JPG/PNG 展示预览图而非文件流。
5. **oss_path 变更审计验证**：通过管理后台上传文件迁移/冷热分层功能触发 oss_path 变更，验证审计日志成功记录变更前后的 oss_path 值，且哈希链校验连续未断裂。
